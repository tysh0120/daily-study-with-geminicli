# 2026-06-02: Node.js 堅牢なファイルベース・メッセージキュー

## 1. 背景と目的
実際のシステム開発において、メモリ上のデータはプロセス終了時に消失します。重要なタスクやログを確実に処理するためには、ディスクへの永続化が不可欠です。
本課題では、Node.js のファイルシステム API を駆使し、**「クラッシュしてもデータが壊れない、かつ効率的な」**ファイルベースのメッセージキューを構築します。

また、本日から **PR（Pull Request）ベースのワークフロー**を導入します。これは、実務におけるコードレビューと品質管理のプロセスを擬似的に体験するためのものです。

## 3. 課題要件

### 機能要件
- [ ] **enqueue(message)**: メッセージをキューの末尾に追加する。
    - メッセージは JSON オブジェクトとする。
    - 追記型（Append-only）のログファイルを使用し、効率を確保する。
- [ ] **dequeue()**: キューの先頭からメッセージを 1 つ取り出す。
    - 取り出したメッセージはキューから削除される。
    - 削除操作は「アトミック」である必要がある（クラッシュ時にデータが重複したり消失したりしない）。
- [ ] **peek()**: 先頭のメッセージを確認する（削除はしない）。
- [ ] **size()**: 現在のメッセージ数を取得する。
- [ ] **recovery()**: プロセス起動時に、前回の未完了操作をクリーンアップして一貫性を復元する。

### 非機能要件
- **原子性 (Atomicity)**: 書き込み中にプロセスが強制終了しても、ファイルが壊れないこと。
- **排他制御**: 複数のインスタンスが同じディレクトリを操作しようとした場合、適切にロックをかけるかエラーを出す（オプショナルだがあれば尚良し）。
- **効率**: キューが大きくなっても、`enqueue` のパフォーマンスが極端に低下しないこと。

### 推奨される実装方針
- メッセージごとにファイルを分けるのではなく、1つのログファイル（`queue.log`）と、現在の読み取り位置を管理するファイル（`manifest.json`）を分ける方式。
- あるいは、各メッセージを1つの `.msg` ファイルとして保存し、連番で管理する方式。

## 5. ステップアップ・ヒント
- **Locking**: `fs.mkdir` や `fs.open` の `x` フラグを利用して簡易的なロックファイルを実装できます。
- **Atomic Write**: ファイルを直接上書きするのではなく、一時ファイルに書き込んでから `fs.rename` すると、OS レベルで原子性が保証されます。

  1. 原子性（Atomicity）を保つファイル書き込み
  例えば、管理ファイル（manifest.json）を更新する際：
   1. 直接 manifest.json を開いて書き込む
      → NG: 書き込み中に止まると、ファイルが空になったり壊れたりします。
   2. 一時ファイル（manifest.json.tmp）に書き込み、完了後に fs.rename する
      → OK: rename は OS レベルで一瞬（アトミック）に行われるため、「古い状態」か「新しい状態」のどちらかしか存在しません。

  2. Dequeue（取り出し）の整合性
  メッセージを取り出す（削除する）際：
   1. メッセージファイルを削除する。
   2.  管理ファイルを更新して「取り出し済み」とする。

  もし 1 と 2 の間で止まったらどうなるでしょうか？
   - 次回起動時に「ファイルはないのに、管理上は未取り出し」となってしまい、エラーになるかもしれません。
   - 逆に 2 を先にして 1 の前に止まると、同じメッセージを2回読み取ってしまう（二重取り出し）可能性があります。
